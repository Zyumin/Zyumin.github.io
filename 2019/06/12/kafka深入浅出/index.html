为什么选择kafka

  • 多个生产者
  • 多个消费者
  • 基于磁盘的数据存储
  • 伸缩性
  • 高性能

硬件选择

  • 磁盘吞吐量
  • 磁盘容量
  • 内存
  • 网络
  • CPU

磁盘容量和网络是瓶颈

集群

Topic主题:一个主题可以有多个Partition分区

生产者

  1. ProducerRecord对象需要包含目标主题和内容 (可以指定键和分区)
  2. 在发送时 先序列化成字节数组
  3. 数据被传给分区器(根据键,如果没有则使用轮询算法 随机) 添加到记录批次,
  4. 最后独立线程负责把这些记录批次发送到相应broker上

发送方式:

  • 异步发送
  • 同步发送
  • 异步回调方式

确认接收ACK机制

创建消费者

订阅主题

轮询 (群组协调、分区再均衡、发送心跳、获取数据)

​ 轮询不只获取数据那么简单,第一次调用新消费者的poll方法 它会负责查找GroupCoordinator,加入群组,接受分配的分区

提交和偏移量(再均衡时会被重新处理)

​ 自动提交,会重复处理消息 ,每隔5s, 消费者会自动从poll方法接收到最大偏移量提交上去

​ 同步提交

​ 异步提交

再均衡监听

深入Kafka

集群成员关系

用zookeeper来管理集群

控制器 broker

复制

​ 每个主题被分为若干个分区,每个分区有多个副本(每个broker可以有多个副本)

​ 每个分区 都有一个首领副本 所有生产者 消费者请求都会经过这个副本 和一个首选首领

​ 跟随者副本

处理请求

​ 生产请求 写入成功 acks = 0 or 1 or all

​ 获取请求 零复制 broker延迟作出响应以便积累足够的数据

​ 其他 元数据

物理存储

分区分配

​ 分区是一个基本存储单元

文件管理

​ kafka的数据不会一直保留,分区分成若干个片段

​ 活跃片段永远不会被删除

文件格式

​ 消息、偏移量 、压缩

索引

​ 每个分区维护一个索引 索引把偏移量映射到片段文件和偏移量在文件里的位置

清理

​ delete和compact策略

​ 干净数据 清理过的

​ 污浊数据 清理后新写入的

可靠的数据传递

可靠性保证

保证分区顺序

只有当消息写入所有同步副本.(可以选择接受不同类型的确认)

只要还有一个副本是活跃的,那么已提交的消息就不会丢失了

消费者只能读取已经提交的消息

同步副本

  • 活跃会话

  • 从首领获取过最新消息

非同步副本

broker配置
